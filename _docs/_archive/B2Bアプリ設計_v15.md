## M-1: MCID根拠に教育行動科学文献を追加

**課題:** Norman et al. (2003) は臨床指標（HRQoL）の文献。streakRateは行動指標なので、教育・行動科学領域の先行研究を主根拠に追加すべき。

**修正内容:** `lib/phaseRollbackSop.ts` の `ENCOURAGEMENT_EFFECTIVENESS_THRESHOLDS` を更新。

```typescript
export const ENCOURAGEMENT_EFFECTIVENESS_THRESHOLDS = {
  streakRecovery: {
    threshold: 0.10,

    // 根拠1: 教育行動科学（主根拠）
    // Lally et al. (2010) "How are habits formed" の習慣形成研究では、
    // 行動の一貫性変化は個人差が大きく（18〜254日）、
    // 短期介入（7日）での有意な回復には「安定性の変動幅を超える変化」が必要。
    // streakRateの±10%ポイントは、習慣形成段階（Phase1〜2相当）での
    // 「週次行動一貫性の実質的変化」として設定。
    //
    // 根拠2: 行動変容の効果サイズ（補強根拠）
    // Cohen (1988) の効果量基準では、小効果 d=0.2 を行動指標に適用すると、
    // streakRate（0〜1スケール）での最小意味ある差は推定 0.08〜0.12 程度。
    // 0.10は小効果量の中央値として妥当。
    //
    // 根拠3: 臨床計量学（補強根拠、v14からの継続）
    // Norman et al. (2003) MCID理論（ハーフSD法）
    //
    // 参照:
    // - Lally et al. (2010) "How are habits formed: Modelling habit formation in the real world"
    // - Cohen (1988) "Statistical Power Analysis for the Behavioral Sciences"
    // - Norman et al. (2003) "Interpretation of changes in health-related quality of life"

    pilotCalibration: {
      // v14から継続（変更なし）
      milestone: 'M3（2026-07）',
      dataRequirement: '励ましメッセージ送信50件以上',
      method: 'ROC曲線分析でstreakRate改善の最適分離点を特定',
      acceptanceCriteria: 'AUC > 0.65',
      adjustmentRange: '0.05〜0.20の間で再設定',
      fallback: 'M3条件未達の場合は0.10を維持してM6で再評価',
    },
  },
}
```

## M-2: エスカレーション先のper-org設定

**課題:** エスカレーション先が `org_admin` 固定で、スクールカウンセラー・学年主任等への対応が不可。

**修正内容:** `org_escalation_settings` テーブル追加 + `SOP_ESCALATION_POLICY` 更新。

### DB追加

```sql
CREATE TABLE org_escalation_settings (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id UUID NOT NULL UNIQUE REFERENCES classroom_orgs(id),
  escalation_targets JSONB NOT NULL DEFAULT '["org_admin"]',
  -- 設定可能な値: 'org_admin' | 'school_counselor' | 'grade_supervisor' | 'platform_admin'
  escalation_emails JSONB DEFAULT '[]',
  -- org_admin以外の追加メールアドレス（スクールカウンセラー等）
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
-- RLS: org管理者のみ閲覧・更新可
```

### `lib/phaseRollbackSop.ts` 更新

```typescript
export const SOP_ESCALATION_POLICY = {
  unresolvedEscalation: {
    thresholdDays: 14,
    escalationTargetResolution: {
      method: 'org_escalation_settings テーブルを参照',
      fallback: 'org_escalation_settingsが未設定の場合は org_admin → platform_admin の順',
      defaultTargets: ['org_admin'],
    },
  },
}
```

## M-4: エスカレーション通知のPII保護方針 + RLS

**課題:** 通知内容に生徒氏名・教師氏名が直接埋め込まれており、GDPR リスク。`escalated_at / escalated_resolved_at` の RLS 設定も未明示。

**修正内容:** `lib/phaseRollbackSop.ts` の通知テンプレートをPII最小化 + RLSポリシー明示。

```typescript
export const SOP_ESCALATION_NOTIFICATION_POLICY = {
  piiHandling: {
    emailBody: {
      // メール本文では個人識別情報を最小化
      // 氏名は使用しない → 「担当している生徒」「担当教師」等の匿名表現を使用
      // ダッシュボードURLのみ記載し、詳細はログイン後に確認させる
      template: {
        subject: '【要対応】学習アラートが{days}日間未対応です（MON管理システム）',  // 氏名なし
        body: '担当されている生徒の学習アラートが{days}日間未解決のままです。\n\n詳細はMON管理システムよりご確認ください:\n{dashboardUrl}\n\nこのメールは自動送信されています。',
      },
      rationale: 'GDPR Article 5(1)(c): データ最小化原則。メール転送・誤送信時の個人情報漏洩リスクを排除。',
    },
    inAppNotification: {
      // アプリ内通知は認証済みユーザーへの表示のため、氏名を含めてよい
      template: '{teacherName}先生が担当する{studentName}さんの{triggerType}アラートが{days}日間未解決です。',
    },
  },

  // RLS設定の明示
  rlsForEscalationFields: {
    table: 'teacher_sop_notifications',
    escalated_at: 'org_admin以上のロールのみ参照可（教師は自身のstudentの通知のみ）',
    escalated_resolved_at: '同上',
    implementation: `
      -- teacher_sop_notificationsのRLSポリシー更新
      -- 通常ポリシー: teacher_id = auth.uid()
      -- 管理者ポリシー: auth.jwt()->>'role' IN ('org_admin', 'platform_admin')
      CREATE POLICY "org_admin_can_see_all_sop_notifications"
        ON teacher_sop_notifications FOR SELECT
        USING (
          auth.jwt()->>'role' IN ('org_admin', 'platform_admin')
          OR teacher_id = auth.uid()
        );
    `,
  },
}
```

## J-1: streak_rateの定義・精度・6日ラグの許容設計

**課題:** 分母・分子の定義、小数点精度、週次Cron依存による最大6日ラグを許容設計として明記すべき。

**修正内容:** `lib/phaseRollbackSop.ts` に `STREAK_RATE_DEFINITION` を追加。

```typescript
export const STREAK_RATE_DEFINITION = {
  formula: 'streak_rate = 連続学習セッション数 / 直近30日の営業日数（土日除く = 最大22日）',
  precision: 'FLOAT（PostgreSQL）= 小数点以下7桁精度。0.20閾値は厳密比較（>=, <）',

  updateFrequency: '週次（毎週日曜）',
  maxLag: '最大6日（月曜〜土曜に閾値を超えた場合、翌日曜まで未検知）',
  lagAcceptance: {
    accepted: true,
    rationale: 'streak_rate低下は数週間単位のトレンドを対象としており、1週間以内の検知ラグは介入タイミングに実質的影響なし。Lally (2010) の習慣形成研究でも短期変動は測定ノイズとして扱う。',
    mitigationIfNeeded: '将来的に日次Cronへの昇格が必要な場合は、毎時のescalate-assignmentsジョブへの統合を検討（現在はVercel Pro 20ジョブ中余裕9スロット）',
  },
}
```

## J-5: reason=NULLの監査ポリシー定義

**課題:** `teacher_consent_override_logs` の `reason=NULL` が監査時にどう扱われるかのポリシーが未定義。GDPR Article 5(2) リスク。

**修正内容:** `infrastructure/deployment-checklist.ts` の `AUDIT_LOG_POLICIES` に追記。

```typescript
export const AUDIT_LOG_POLICIES = {
  teacher_consent_override_logs: {
    reasonNullPolicy: {
      meaning: 'NULL = 理由が入力されなかった（非記録）',
      gdprCompliance: `
        GDPR Article 5(2) 説明責任原則への対応:
        - NULLレコードは「理由を記録しない選択をした」という事実自体が監査証跡。
        - confirmed_text（確認ダイアログ文言）が必須記録されているため、
          「何を承諾したか」の証拠は保持される。
        - 理由なしでの操作は技術的には可能だが、UIのソフトウォーニングにより自発的記録を促す。
        - 監査時: reason IS NULL のレコードは「理由未記録」として別途カウント・報告可能。
      `,
      auditQueryExample: `
        -- 監査レポート用クエリ（reason記録率の確認）
        SELECT
          action_type,
          COUNT(*) as total,
          COUNT(reason) as with_reason,
          ROUND(COUNT(reason)::numeric / COUNT(*) * 100, 1) as reason_rate
        FROM teacher_consent_override_logs
        GROUP BY action_type;
      `,
    },
  },
}
```
