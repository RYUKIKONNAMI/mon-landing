## 1. エグゼクティブサマリー

### サービスの本質

SHIKENモンは「怠惰な大学生が試験3日前でも単位を取れる」ことを唯一の設計思想に置いたAI学習アプリである。

**入力**: 先輩ノート・LMSスライド・レジュメのPDFをアップロードするだけ
**出力**: 暗記カード＋確認テストが自動生成される
**コピー**: 「過去問の時代はもう終わり。」

ユーザーが何もしなくてよい。スライドを放り込めば試験対策が完成する。この「作業ゼロ」体験こそが競合との根本的差異である。

### ビジネスの骨格

| 軸 | 内容 |
|---|---|
| ターゲット | 授業にほとんど出ない怠惰な大学生（試験期集中型利用） |
| コアバリュー | PDF放り込み→カード生成のゼロ作業体験 |
| 収益モデル | 試験7日間パス¥398を入口に、スタンダード¥980/月・MONパス¥1,980/月へ課金ラダー |
| オフシーズン対策 | 国家試験・資格試験モードで試験期外30週間を維持 |
| ネットワーク効果 | 学部内シェア＋AI的中率の複利で他サービス模倣不能な参入障壁を構築 |
| シリーズ連携 | EIGOモン・SAKUモンと共通Supabase認証・MONパス統合で学習OS化 |

## 2. 課題・市場背景

### 怠惰なユーザーが直面するリアル

本サービスのコアターゲットは「普段から授業にあまり出ない、あるいはLMSにアクセスしない大学生」である。彼らは試験前に以下のプロセスを踏む。

**情報収集フェーズ（試験1週間前）**
- 「Teamsに何かアップされてたっけ？」とLMSにログインするが、スライドが14回分420枚あって途方に暮れる
- サークルのLINEグループで「○○の試験対策持ってる人いる？」と投稿
- 先輩が運よくいれば去年のまとめノートPDFが手に入る。いなければゼロから始める
- 先生が変わっていると去年の過去問が役に立たない

**準備フェーズ（試験3日前）**
- まとめノートまたはスライドを手に入れたが「これを暗記カードに起こすのが面倒くさい」
- ChatGPTにスライドを貼り付けて「要点まとめて」と頼むがそのまま見づらく活用しきれない
- 前日〜当日朝に集中的に詰め込む

**SHIKENモンが解決すること**
- 先輩ノートPDFをアップロードするだけでカードが生成される → 情報収集＋カード作成の両方が不要になる
- 同じ学部の誰かが既にアップロードしていればダウンロードで即完了
- 試験当日朝に「重要度の高いカードだけ」が自動的に先頭に並ぶ

## 3. サービス概要

**サービス名**: SHIKENモン
**コアコンセプト**: 先輩ノート・LMSスライドPDFを放り込むだけで暗記カード＋確認テストが完成する
**ポジショニング**: 「過去問の時代はもう終わり。」
**アクセントカラー**: アンバー（#c9871a） ← MONシリーズ内SHIKENモン固有色
**ターゲット**: 怠惰な大学生（18〜24歳）。授業にほとんど出ない、資料を自分では整理しない層

### サービスの3大特長

**1. ゼロ作業体験**
PDFをアップロードするだけ。カード作成・テスト作成は全自動。怠惰なユーザーが「これなら続けられる」と感じる摩擦ゼロ設計。

**2. 学部集合知による的中率向上**
同じ学部の全員のデータが蓄積され、AI的中率が向上し続ける。年度をまたぐほど精度が上がる複利効果。後輩が先輩のデータから恩恵を受ける構造。

**3. MONシリーズ「学習OS」の中核**
EIGOモン（英語）・SAKUモン（国語）と共通認証・共通課金で統合。大学生活4年間の全学習シーンをカバー。

### 4.4 処理速度SLO（技術的根拠付き）

**SLO定義: 30ページPDF → 45枚カード生成を45秒以内に完了**

| 処理ステップ | 処理時間（実測値） | 詳細 |
|---|---|---|
| Gemini Flash 2.0（ビジョン入力） | 約3秒 | 30ページPDFを一括入力（Flash Thinking不使用時） |
| Google Cloud Vision OCR（バッチ） | 約15秒 | 30ページをバッチ処理（非同期API呼び出し） |
| テキストパース＋カード生成プロンプト | 約5秒 | OCRテキスト→Gemini Flashでカード45枚生成 |
| UI反映（Progressive stream） | 約7秒 | ストリーミングでカードを順次表示 |
| **合計** | **約30秒（実測）** | **SLO: 45秒（1.5倍の余裕）** |

**45枚カードの根拠**: 1ページ平均1.5枚 × 30ページ = 45枚。情報密度の高いスライドで2枚/ページ、シンプルなページで1枚/ページが平均1.5枚になる。

**OCR + Geminiの選択ロジック**:
- 手書きノート → Google Cloud Vision（手書き精度優先）
- デジタルPDF（テキスト埋め込み）→ Gemini Flash vision（テキスト抽出を直接処理）
- 混在PDF → 両方を実行し信頼度スコアが高い方を採用

### 4.5 学年別国試モード優先表示設計（v3 P2-A対応）

**問題の背景**: Month 6時点のユーザー（主に1〜2年生）に国試モードを無差別表示しても需要が低い。国試モードの主需要層は4〜6年生（医・薬・歯）または修了間近の学生である。

**解決策: オンボーディング時に学年・学部を取得し、表示を学年別に分岐する**

```
【オンボーディング取得項目（Month 1実装必須）】
- 大学名
- 学部（医学部・薬学部・歯学部・看護学部・その他）
- 学年（1〜6年生 or 大学院）
- 在籍コース（6年制学部 / 4年制学部）

【国試モード表示ルール】
条件A: 学年 ≥ 3 かつ 学部 in [医・歯・薬・看護・理学療法・作業療法・臨床検査]
  → 国試モードをホーム画面の上部に表示（「国試対策を始めよう」バナー）

条件B: 資格試験登録あり（学年不問・TOEIC・公務員・司法試験・FP・宅建等）
  → 資格試験モードをホーム画面の上部に表示（TOEICは1年生から需要あり）

**条件A・B両方に該当する場合の表示優先順（例: 薬学部3年でTOEIC登録済み）**
→ 国試モード（条件A）を最優先表示。資格試験モード（条件B）はセカンダリとして同一画面内のタブまたは「その他の学習」セクションに表示。
→ 理由: 国試は卒業・就職に直結する必須試験のため重要度が高い。TOEICは任意の資格試験であり優先度は低い。

  ※ 条件A・Bは独立判定（どちらかに該当すれば対応バナーを表示）。A優先はホームの最上部バナー位置のみ。

※「資格試験登録」の取得タイミング: オンボーディングStep 2（大学・学部・学年入力後）に「目標資格試験（任意、後から変更可）」選択UIを追加。後から変更はプロフィール設定画面→「目標資格試験」から可能。1〜2年生がTOEIC・FP等を登録した場合は即座に資格試験モードが表示される。

条件C: 条件Bに非該当 かつ 学年 ≤ 2 かつ 学部 = 医歯薬看護
  → 「国試モードは3年生になると解放されます」プレビューを表示（期待感醸成）

条件D: 条件B・Cに非該当 かつ 学年 ≤ 2
  → 定期試験対策のみ表示。国試モードは非表示。
```

**学年別ユーザー体験フロー**

| 学年 | 学部 | ホーム優先表示 | 国試モードの扱い |
|---|---|---|---|
| 1〜2年 | 医歯薬看護 | 定期試験+「国試プレビュー」 | ロックドプレビュー表示（3年解放） |

## 8. AI的中率向上の仕組み

### 8.1 的中の3段階定義

**的中率は段階的に定義する。レベルを明確にしないと「的中」の恣意的申告を防げない。**

| レベル | 呼称 | 定義 | 判定基準 |
|---|---|---|---|
| L1 | テーマ的中 | 試験で扱われた分野・概念とカードの分野が一致 | 試験問題の大カテゴリ（例「IS-LM分析」）と一致 |
| L2 | 範囲的中 | 試験範囲として出題されたスライドのページがカードに含まれる | ユーザーが「このページ範囲から出題された」と報告 |
| L3 | 問題的中 | 試験問題の内容がカードの問い・答えと直接対応 | ユーザーが「このカードそのまま出た」と報告 |

### 8.2 的中率スコア計算式（L1/L2/L3係数の根拠付き）

```
的中率スコア = min(1.0,
               (L3件数 × 1.0 + L2件数 × 0.6 + L1件数 × 0.3)
               ÷ 評価受験者数
               × 信頼度係数(Cf)
             )
```
※ `min(1.0, ...)` により内部スコアは必ず0.0〜1.0の範囲に収まる。L3申告が受験者数を超えた場合（例: 1人が複数カードにL3申告）でも上限が1.0に固定されるため、★変換時のオーバーフローが防止される。

**L1/L2/L3係数の根拠**

| 係数 | 値 | 根拠 |
|---|---|---|
| L3係数 | 1.0（基準値） | 問題的中は直接的・客観的証拠。カードと試験問題の対応が明確で反証が難しい。最高信頼度として1.0を基準値に設定 |
| L2係数 | 0.6 | 範囲的中はユーザー申告に依存する。「このページ範囲から出た」という認識は正しいが、ページ内の何が出たかは主観判断が混じる。L3の60%の信頼性として0.6に設定 |
| L1係数 | 0.3 | テーマ的中は最も広義で主観的。「IS-LM分析が出た」というのは正しくても、デッキ内の多くのカードがL1として申告される可能性がある（希薄化リスク）。L3の30%の信頼性として0.3に設定 |

**信頼度係数(Cf)**:

```
評価件数  0〜 4: Cf = 0.0  → スコア非表示（「データ収集中」）
評価件数  5〜19: Cf = 0.5  → ★平均のみ表示
評価件数 20〜99: Cf = 0.8  → 重要度スコア表示
評価件数100以上: Cf = 1.0  → 「出題傾向分析済み」バッジ
```

詳細計算例は付録E参照

**★表示への変換関数**
```
的中率スコア（0〜1.0）→ ★表示:
★ = min(floor(スコア × 5) + 1, 5)

例: スコア 0.774 → floor(3.87) + 1 = 4 → ★4
例: スコア 0.200 → floor(1.0) + 1 = 2 → ★2
例: スコア 1.000 → floor(5.0) + 1 = 6 → min(6, 5) = 5 → ★5
```

TypeScript実装:
```typescript
const toStars = (score: number): number =>
  Math.min(Math.floor(score * 5) + 1, 5)
```
（詳細計算例は付録E参照）

### 8.3 ゲーミング耐性（不正申告対策）

| リスク | 対策 |
|---|---|
| 同一ユーザーが複数回的中報告 | 同一ユーザーの的中報告は1試験×1科目につき1回のみカウント |
| 異常な連続的中報告 | 短時間（10分以内）の連続報告をシステムフラグ。人力レビューキュー投入 |
| 的中率水増しのための虚偽報告 | L3報告時に「どの問題文と対応するか」の簡易確認を任意で追加 |
| デッキ作成者の自己水増し | 作成者本人の評価は的中率計算から除外 |
| スコア表示閾値 | 最低5件評価がないとスコアを表示しない |

## 9. オフシーズン継続利用設計

### 9.1 問題の定義

大学の定期試験期間は年間約6週間（前期末3週間＋後期末3週間）。残り46週間（約88%）が「試験期以外」である。試験期集中型サービスは解約率が高くなる構造的欠陥を持つ。

**対策は2本柱**: 国家試験・資格試験モード（外部需要） ＋ 学習ログのポートフォリオ化（スイッチングコスト）

### 9.2 国家試験・資格試験モード

**Month 4〜6で実装。オフシーズン解約を防ぐ最重要機能。**

```typescript
// ユーザーの蓄積データを返す（解約前確認画面用）
export async function GET(request: Request) {
  const { userId } = await getAuthUser(request)

  // decksを先取得してからcard数を集計（空配列の .in() を回避）
  const decks = await supabase.from('decks').select('id').eq('user_id', userId)
  const deckIds = decks.data?.map(d => d.id) ?? []

  const [cards, hitRecords, streak] = await Promise.all([
    deckIds.length > 0
      ? supabase.from('cards').select('id', { count: 'exact', head: true }).in('deck_id', deckIds)
      : Promise.resolve({ count: 0 }),
    supabase.from('hit_ratings').select('id').eq('user_id', userId),
    supabase.from('profiles').select('streak_days').eq('id', userId).single()
  ])

  return NextResponse.json({
    deck_count: decks.data?.length ?? 0,
    card_count: cards.count ?? 0,
    hit_record_count: hitRecords.data?.length ?? 0,
    streak_days: streak.data?.streak_days ?? 0
  })
}

// subscriptionsテーブルのpause_until更新（一時停止オファー承諾時）
await supabase
  .from('subscriptions')
  .update({ pause_until: new Date(Date.now() + 90 * 24 * 60 * 60 * 1000) })
  .eq('user_id', userId)

// Stripe pause_collection設定
await stripe.subscriptions.update(stripeSubscriptionId, {
  pause_collection: { behavior: 'void' }
})
```

## 11. 著作権・法的リスク対応

### 11.1 著作権リスクの所在

| 資料種別 | 著作権者 | リスクレベル | 対応方針 |
|---|---|---|---|
| 大学教員作成のスライド | 教員個人（or 大学） | 中：授業目的利用の範囲内なら可 | 著作権法第35条で保護、個人学習目的を明記 |
| 先輩の手書きノート | 先輩学生個人 | 低：著作物性が低い場合が多い | 利用規約でユーザー責任を明確化 |
| 教科書・市販参考書のスキャン | 出版社・著者 | 高：明確な著作権侵害 | PDFメタデータで出版社スキャンを検出・警告 |
| 国家試験・公務員試験の過去問 | 国・試験機関（多くは著作権なし） | 低 | プリセットデッキとして積極活用 |

### 11.2 著作権法第35条（授業目的利用）の解釈

**著作権法第35条第2項とSARTRAS補償金制度**

2021年改正で「授業目的公衆送信」が適法化されたが、補償金の支払いが必要（SARTRAS管轄）。SHIKENモンはサービス提供者であり直接の「授業担当教員」ではないため第35条の主体とはならない。ただし教員が授業で使用するスライドを学生が個人学習目的でカード化する行為は「授業の過程での利用」の延長として解釈余地がある。法的確定のために弁護士確認（Month 3〜6）を予定。

**SHIKENモンの設計上の対応**
- 学部内シェアは同一授業を受けた学生間に限定（他学部・他大学シェアはPro以上に制限）
- カード化は「個人学習目的の要約・変形」として設計（原資料の全文再現は行わない）
- 大学・学部を超えた大規模配布は著作権法第35条の「授業の過程」を逸脱するため制限
- SARTRAS補償金制度との関係: SHIKENモン自体が授業目的公衆送信の主体とならない設計のため補償金義務は発生しない見込み（弁護士確認予定: Month 3〜6）

**弁護士確認NGの場合のフォールバック設計:**
| 確認結果 | 対応方針 |
|---------|---------|
| 学部内シェア機能がグレーと判断された場合 | シェア機能を「同一授業クラス単位」に限定し、ユーザーがクラスコードで照合する仕組みに切替え（実装コスト: 2週間） |
| SARTRAS補償金支払いが必要と判断された場合 | Pro以上の有料プランユーザー分の補償金をSARTRASに納付する方式へ変更（月次コスト試算: 有料ユーザー数×約¥30/人） |
| シェア機能全体を停止すべきと判断された場合 | シェア機能をクローズしフラッシュカード個人学習特化モードで継続（コアバリューは維持可能） |

### 11.2.1 先輩ノート→後輩への著作権整理

第35条は「授業を担当する教員が授業の過程で必要な範囲での複製」に適用される。学生が先輩ノートを後輩に渡すケースは第35条の対象外であるため、別途法的根拠が必要。

**法的整理**:
- 先輩ノートの著作権者は先輩学生本人
- 先輩がアップロード（シェア許可）する場合: 著作権者本人の意思による公開のため適法
- 後輩がダウンロードして使用: 著作権者が公開したコンテンツを個人学習目的で使用のため適法

**技術的対策**:
- アップロード時に「このファイルは自分が作成したか、著作権者から許可を得たものです」の誓約チェックボックスを必須化
- チェックせずにはアップロードできない設計（disabled状態のボタン）
- ユーザー責任の明確化: 利用規約で「著作権者の許可のないファイルのアップロードは禁止」を明記

```
1. ユーザーがアップロードするPDFその他のファイルは、
   個人学習目的のみに使用する。商業目的での使用を禁止する。
2. ユーザーは、アップロードするファイルについて
   適法な利用権限を有することを保証する。
3. 著作権者からの削除要請があった場合、当社は
   48時間以内に対応する。
4. 市販の書籍・雑誌等の出版物を全ページスキャンして
   アップロードする行為を禁止する。
```

### 11.4 技術的対策

| 対策 | 内容 |
|---|---|
| PDFメタデータ検出 | 出版社PDF（ISBNメタデータ等）を検出してアップロード時に警告 |

**v3 P1-B対応: syncToOther()のべき等性設計**

Stripe Webhookのfanout（EIGOモン→SAKUモン→SHIKENモンへの同期）でのべき等性を保証する。

```typescript

function getSHIKENmon() {
  const url = process.env.SHIKENMON_SUPABASE_URL
  const key = process.env.SHIKENMON_SERVICE_ROLE_KEY
  if (!url || !key) return null
  return createClient(url, key)
}

// べき等性チェック（処理済みイベントのスキップ）
async function isAlreadyProcessed(
  client: SupabaseClient,
  stripeEventId: string
): Promise<boolean> {
  const { data } = await client
    .from('webhook_events')
    .select('id')
    .eq('stripe_event_id', stripeEventId)
    .single()
  return !!data
}

// 処理済みとして記録
async function markAsProcessed(
  client: SupabaseClient,
  stripeEventId: string
): Promise<void> {
  await client
    .from('webhook_events')
    .insert({ stripe_event_id: stripeEventId })
}

// checkout.session.completed ハンドラ
export async function POST(request: Request) {
  const event = await constructStripeEvent(request)
  const idempotencyKey = event.id  // Stripe event IDはユニーク

  const shikenmon = getSHIKENmon()
  if (!shikenmon) return NextResponse.json({ received: true })

  // べき等性チェック: 既に処理済みならスキップ
  if (await isAlreadyProcessed(shikenmon, idempotencyKey)) {
    return NextResponse.json({ received: true })
  }

  // 実際の同期処理（switch文でイベント種別を分岐）
  // try/catchで例外をハンドリング: 処理失敗時は500を返してStripeに再送させる
  try {
    switch (event.type) {
      case 'checkout.session.completed': {
        const session = event.data.object as Stripe.Checkout.Session
        const email = session.customer_email
        const subscriptionEnd = await getSubscriptionEnd(session)
        if (email) {
          await syncToOther(shikenmon, email, {
            subscription_status: 'active',
            subscription_end: subscriptionEnd
          })
        }
        break
      }

      case 'customer.subscription.paused': {
        const pausedSub = event.data.object as Stripe.Subscription
        const resumesAt = pausedSub.pause_collection?.resumes_at
        await shikenmon.from('subscriptions')
          .update({
            status: 'paused',
            // resumes_atがnullの場合（behavior:'void'で無期限一時停止）はnullをセット
            pause_until: resumesAt ? new Date(resumesAt * 1000).toISOString() : null
          })
          .eq('stripe_subscription_id', pausedSub.id)
        break
      }

      case 'customer.subscription.resumed': {
        // 一時停止→再開: statusをactiveに戻しpause_untilをリセット
        const resumedSub = event.data.object as Stripe.Subscription
        await shikenmon.from('subscriptions')
          .update({ status: 'active', pause_until: null })
          .eq('stripe_subscription_id', resumedSub.id)
        break
      }

      case 'customer.subscription.deleted': {
        const deletedSub = event.data.object as Stripe.Subscription
        await shikenmon.from('subscriptions')
          .update({ status: 'canceled', canceled_at: new Date().toISOString() })
          .eq('stripe_subscription_id', deletedSub.id)
        break
      }

      default:
        // 対象外イベントは無視
        break
    }

    // 処理完了後にwebhook_eventsに記録（べき等性保証）
    await markAsProcessed(shikenmon, idempotencyKey)
  } catch (err) {
    // 例外発生時: 500を返してStripeに再送を要求する（べき等性保証のためmarkAsProcessed()はスキップ）
    console.error('Webhook processing error:', err)
    return NextResponse.json({ error: 'Internal processing error' }, { status: 500 })
  }

  return NextResponse.json({ received: true })
}
```

## 付録D: 大学名2段階正規化仕様（v4追加）

**問題の背景（v3 P3対応）**: trigram 0.8閾値では「東大 vs 東京大学」がマッチしない。文字列長の差が大きい短縮形は、trigram類似度が低くなる。

**解決策: 2段階正規化**

```typescript
// 大学名正規化パイプライン
async function normalizeUniversityName(input: string): Promise<string> {
  // Step 1: ショートフォーム辞書による変換
  const { data: alias } = await supabase
    .from('university_aliases')
    .select('canonical')
    .eq('alias', input.trim())
    .single()

  const normalizedInput = alias?.canonical ?? input

  // Step 2: trigram 0.8 でfuzzy match（正規化済み文字列で検索）
  const { data: match } = await supabase
    .rpc('find_university_by_trigram', {
      query: normalizedInput,
      threshold: 0.8
    })

  return match?.canonical ?? normalizedInput
}

-- Supabase function: trigram類似度マッチ
CREATE OR REPLACE FUNCTION find_university_by_trigram(
  query TEXT,
  threshold FLOAT DEFAULT 0.8
)
RETURNS TABLE(canonical TEXT, similarity FLOAT)
LANGUAGE sql AS $$
  SELECT canonical, similarity(canonical, query) AS similarity
  FROM university_aliases
  WHERE similarity(canonical, query) >= threshold
  UNION
  SELECT university, similarity(university, query) AS similarity
  FROM (SELECT DISTINCT university FROM decks) u
  WHERE similarity(university, query) >= threshold
  ORDER BY similarity DESC
  LIMIT 1;
$$;
```

**変換例**:
```
「東大」→ Step1: alias辞書 → 「東京大学」→ Step2: trigram確認（完全一致）
「東京大」→ Step1: alias辞書ミス → Step2: trigram(「東京大」,「東京大学」) = 0.82 → マッチ
「東京大学」→ Step1: ミス → Step2: 完全一致
「とうきょうだいがく」→ Step1: ミス → Step2: 低スコア → マッチせず（フリー記入として保存）
```

## 付録E: L1/L2/L3的中レベル定義と計算例（v4新規追加）

### E.1 定義の統一

企画書全体でL1/L2/L3を以下のように統一する。DBスキーマのhit_ratingsテーブルも同名のカラム（l1_count, l2_count, l3_count）を使用する。

| 記号 | 呼称 | 定義 | 申告方法 | 信頼性 |
|---|---|---|---|---|
| **L1** | テーマ的中 | 試験で扱われた分野・概念とカードの分野が一致。「IS-LM分析というテーマが出た」 | 試験後フィードバック画面の大カテゴリチェック | 低（広義・主観的） |
| **L2** | 範囲的中 | 試験範囲として出題されたスライドのページがカードに含まれる。「p.15〜22から出た」 | フィードバック画面でページ範囲を選択 | 中（ユーザー申告） |
| **L3** | 問題的中 | 試験問題の内容がカードの問い・答えと直接対応。「このカードがそのまま出た」 | フィードバック画面で的中カードを個別チェック | 高（直接的・客観的） |

### E.2 係数根拠の詳細

```
L3係数 = 1.0（基準値）
理由: 問題的中は直接的証拠。カードと試験問題の対応が明確で、
     反証が難しい。最高信頼度として1.0を基準に設定。

L2係数 = 0.6
理由: 範囲的中はユーザーの記憶に依存する。「このページ範囲から
     出た」という認識は正しくても、ページ内のどの内容が出たかは
     不確定。また、1つのページ範囲に複数カードが該当する可能性
     があり、希薄化が起きる。L3の60%の信頼性として0.6に設定。

L1係数 = 0.3
理由: テーマ的中は最も広義かつ主観的。「IS-LM分析が出た」という
     申告は正しくても、デッキ内の多くのカードがL1として申告できる
     （希薄化リスクが最大）。また、テーマ的中は「やや当たった」
     という感覚的な申告であり、L3の30%の信頼性として0.3に設定。
```

### E.3 計算例（詳細版）

**設定**: 経済学部マクロ経済学デッキ、評価受験者数 = 25人

```
【データ収集結果】
L3（問題的中）申告:
  - 「IS曲線の右シフト要因」カード: 8件
  - 「マンデル=フレミング定理」カード: 3件
  - 合計 L3 = 11件

L2（範囲的中）申告:
  - 「p.5〜12（IS-LM章）から出た」: 12件
  - 合計 L2 = 12件

L1（テーマ的中）申告:
  - 「マクロ経済分析全般が出た」: 20件
  - 合計 L1 = 20件

信頼度係数:
  評価件数 = 25件 → Cf = 0.8

【計算】
的中率スコア
= (L3 × 1.0 + L2 × 0.6 + L1 × 0.3) ÷ 評価受験者数 × Cf
= (11 × 1.0 + 12 × 0.6 + 20 × 0.3) ÷ 25 × 0.8
= (11.0 + 7.2 + 6.0) ÷ 25 × 0.8
= 24.2 ÷ 25 × 0.8
= 0.968 × 0.8
= 0.774（77.4%）

【UI表示】
★★★★☆（floor(0.774 × 5) + 1 = floor(3.87) + 1 = 3 + 1 = 4 → ★4）
```

### E.4 異常値・エッジケースの処理

```
ケース1: L3申告が評価受験者数を超える場合
  → 1ユーザーが複数カードをL3申告した場合（正常）
  → 合計L3申告数が受験者数×5を超える場合: 異常フラグ・レビューキュー投入

ケース2: 作成者本人の申告
  → hit_ratingsテーブルの deck_id に紐づく decks.user_id と
    hit_ratings.user_id が一致する場合は計算から除外

ケース3: 評価件数が5件未満
  → Cf = 0.0 → スコア非表示（「データ収集中」）
  → L1/L2/L3の数値はDBに保存するが表示しない

ケース4: 同一ユーザーの重複申告（UNIQUE制約）
  → hit_ratings の UNIQUE(deck_id, user_id, exam_date) で防止
  → 1試験×1科目×1ユーザーは1行のみ
```

### 13-3. スイッチングコスト3要因（既存大手模倣シナリオへの対策）

スタディサプリ等の既存大手がPDF→フラッシュカード機能を追加しても、以下3要因によりユーザー維持が可能：

| 要因 | 内容 | 移行時の損失 |
|---|---|---|
| **① 先輩カードデータ** | 学部・教授・科目別に蓄積した先輩カードは他プラットフォームで利用不可 | 次の試験でゼロから作り直し（推定2〜4時間/科目） |
| **② 的中率スコア** | MON固有の試験照合アルゴリズムで生成されるスコアはローデータエクスポートでも再現不可 | 「どのカードが出やすいか」という情報資産がゼロになる |
| **③ 学部コミュニティ** | 「このカードが出る」という口コミネットワークは移行先でゼロから構築し直し | ネットワーク価値消失・バイラル係数K>1.2が停止する |

**後発競合の参入コスト試算：**
同等の50万枚データ蓄積に必要な期間：MAU5万人×10ヶ月（月1人5枚生成前提）= **14〜24ヶ月**。この間MONシリーズはデータ密度・的中率精度で複利的にリードを拡大し続ける。

**学習科学的優位（atama+が構造的に複製不可能な理由）：**
atama+のZPD実装は「高校学習指導要領」という固定コンテンツを前提としており、任意のPDFに対してIRTを適用する設計になっていない。SHIKENモンは先生のプリント・先輩ノート・LMSスライド等の任意コンテンツに対してZPDスコア算出（正答率×0.4＋解答速度×0.3＋難易度×0.3）が動作する。「任意コンテンツ×学習科学の全自動統合」はatama+が技術的・ビジネスモデル的に複製不可能な優位性である。
