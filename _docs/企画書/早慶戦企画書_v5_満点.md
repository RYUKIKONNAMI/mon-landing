## 1. エグゼクティブサマリー

「知の早慶戦」は、早稲田大学と慶應義塾大学の学生がEIGOモン・SAKUモンを使って英語読解力で対決する14日間のオンラインイベント。research-aiの第一原理分析で特定された3大リスク（フリーライダー問題、報酬因果関係の曖昧さ、ライバル意識の日常学習への転移の不確実性）を、**小隊制（Squad System）** を核として全て解消する。

---

## 2. 小隊制（Squad System）— 企画の核心

### 2-1. 基本仕様

- **小隊サイズ**: 3〜5人
- **構成制約**: 同大学メンバーのみ（早稲田は早稲田同士、慶應は慶應同士）
- **隊長**: 1人が「隊長」となり招待リンクを発行
- **個人参加**: 小隊に入らなくても個人参加は可能（ただし小隊ボーナスなし）
- **Minimal Mode時**: 2人から結成可能に緩和

### 2-2. 小隊ボーナス（即時性重視）

| ボーナス | 内容 | トリガー |
|---------|------|---------|
| 小隊結成ボーナス | 全メンバーに即時スキャン+3回 | 3人以上で小隊結成完了時 |
| 小隊DPブースト | 小隊全員が当日1スキャン以上 → 小隊DP×1.3倍 | 毎日判定 |
| 連携ボーナス | 小隊内で3人以上が同日にプレイ → 各自+1スキャン翌日付与 | 毎日判定 |
| 上位小隊報酬（終了時） | MONパス無料体験期間延長3ヶ月（各大学上位3小隊） | イベント終了時 |

### 2-3. 小隊制の根拠（なぜ3-5人か）

1. **ソーシャルローフィング研究（Latane et al., 1979）**: 6人超で個人努力が急低下
2. **ゲーム設計**: LoL(5人)、Valorant(5人)、Apex(3人) — 成功事例は3-5人
3. **大学生の友人グループ**: 「いつも一緒のグループ」は典型的に3-5人
4. **最少3人**: 1人離脱しても機能する最小単位
5. **最大5人**: 「誰かがやるだろう」心理の防止

### 2-4. 小隊制がリエンゲージメントの最強装置である証明

通知施策のROI比較（v5.0 DAU分析より）:
- メール通知: 工数3h → DAU +1.3pt → **0.43pt/h**
- 小隊LINE連携促進: 工数0.5h → DAU +5.7pt → **11.4pt/h**（メールの26.5倍効率的）

## 12. データベーススキーマ（新規追加）

### 12-1. squads テーブル

```sql
CREATE TABLE squads (
  id UUID DEFAULT uuid_generate_v4() PRIMARY KEY,
  name TEXT NOT NULL,
  university TEXT NOT NULL CHECK (university IN ('waseda', 'keio')),
  created_by UUID REFERENCES auth.users(id),
  invite_code TEXT UNIQUE NOT NULL,
  event_id TEXT DEFAULT 'war_01',
  created_at TIMESTAMPTZ DEFAULT now()
);

ALTER TABLE squads ENABLE ROW LEVEL SECURITY;
CREATE POLICY "squads_select" ON squads FOR SELECT USING (true);
CREATE POLICY "squads_insert" ON squads FOR INSERT WITH CHECK (auth.uid() = created_by);
```

### 12-2. squad_members テーブル

```sql
CREATE TABLE squad_members (
  id UUID DEFAULT uuid_generate_v4() PRIMARY KEY,
  squad_id UUID REFERENCES squads(id) ON DELETE CASCADE,
  user_id UUID REFERENCES auth.users(id),
  joined_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(squad_id, user_id)
);

ALTER TABLE squad_members ENABLE ROW LEVEL SECURITY;
CREATE POLICY "squad_members_select" ON squad_members FOR SELECT USING (true);
CREATE POLICY "squad_members_insert" ON squad_members FOR INSERT WITH CHECK (auth.uid() = user_id);
```

### 12-3. 既存テーブルへの変更

```sql
ALTER TABLE profiles ADD COLUMN IF NOT EXISTS war_badges JSONB DEFAULT '[]';
ALTER TABLE usage ADD COLUMN IF NOT EXISTS share_count INTEGER DEFAULT 0;
ALTER TABLE usage ADD COLUMN IF NOT EXISTS share_reset_at TIMESTAMPTZ;
```

## 13. API エンドポイント（新規）

### 13-1. 小隊CRUD

**POST /api/squad/create** — 小隊作成（name, userId → squad + 隊長をメンバー登録）

- 小隊DP = 小隊平均ULEスコア × 小隊メンバー数 × 小隊総スキャン数
- 大学DPの既存計算は変更なし

### 13-3. シェアボーナス

**POST /api/share/bonus** — シェア時に+1スキャン付与（1日最大2回）

### 13-4. OGPシェアカード

### 13-5. メール通知

**POST /api/notify/war-report** — Resend APIで全参加者に戦況メール（毎朝8:00）
